LVOCControlMechanism¶

Overview¶

An LVOCControlMechanism is a ControlMechanism that regulates it ControlSignals in order to optimize the performance of the Compositions to which it belongs. It implements a form of the Learned Value of Control model described in Leider et al., which learns to select the value for its control_signals (i.e., its allocation_policy) that maximzes its EVC based on a set of predictors.

Expected Value of Control (EVC)¶

The expected value of control (EVC) is the outcome of executing the composition to which the LVOCControlMechanism belongs under a given allocation_policy, as determined by its objective_mechanism, discounted by the cost under that allocation_policy.

The LVOCControlMechanism’s function learns to predict the outcome of its objective_mechanism from a weighted sum of its predictors, control_signals, interactions among these, and the costs of the control_signals. This is referred to as the “learned value of control,” or LVOC.

Creating an LVOCControlMechanism¶

An LVOCControlMechanism can be created in the same was as any Control Mechanism, with the exception that it cannot be assigned as the controller of a Composition. The following arguments of its constructor are specific to the LVOCControlMechanism:

  • predictors – this takes the place of the standard input_states argument in the constructor for a Mechanism`, and specifies the inputs that it learns to use to determine its allocation_policy in each trial of execution. It can be specified using any of the following, singly or combined in a list:

    • {SHADOW_EXTERNAL_INPUTS: <ORIGIN Mechanism, InputState for one, or list with either or both>} – InputStates of the same shapes as those listed are created on the LVOC, and are connected to the corresponding input_CIM OutputStates by projections. The external input values that are passed through the input_CIM are used as the predictors. If a Mechanism is included in the list, it refers to all of its InputStates.

    • InputState specification – this can be any form of InputState specification that resolves to an OutputState from which the InputState receives a Projection; the value of that OutputState is used as the predictor. Each of these InputStates is marked as internal_only.

    Predictors can also be added to an existing LVOCControlMechanism using its add_predictors method.

  • predictor_function specifies function of the InputState created for each item listed in predictors.

Structure¶

Input¶

An LVOCControlMechanism has one InputState that receives a Projections from its objective_mechanism (its primary InputState <InputState_Primary>`), and additional ones for each of its predictors, as described below.

Predictors¶

Predictors, together with the LVOCControlMechanism’s control_signals, are used by its function to learn to predict the outcome of its objective_mechanism and to determine its allocation_policy.

Predictors can be of two types:

  • Input Predictor – this is a value received as input by an ORIGIN Mechanism in the Composition.
    These are specified in the predictors argument of the LVOCControlMechanism’s constructor (see Creating an LVOCControlMechanism), in a dictionary containing a SHADOW_EXTERNAL_INPUTS entry, the value of which is one or more ORIGIN Mechanisms and/or their InputStates to be shadowed. For each, a Projection is automatically created that parallels (“shadows”) the Projection from the Composition’s InputCIM to the ORIGIN Mechanism, projecting from the same OutputState of the InputCIM to the the InputState of the LVOCControlMechanism assigned to that predictor.
  • Output Predictor – this is the value of an OutputState of some other Mechanism in the
    Composition. These too are specified in the predictors argument of the LVOCControlMechanism’s constructor (see Creating an LVOCControlMechanism), and each is assigned a Projection to the InputState of the LVOCControlMechanism for that predictor.

The current values of the InputStates for the predictors are listed in the predictor_values attribute.

ObjectiveMechanism¶

Like any ControlMechanism, an LVOCControlMechanism receives its input from the OUTCOME OutputState of its objective_mechanism, via a MappingProjection to its primary InputState. By default, the ObjectiveMechanism’s function is a LinearCombination function with its operation attribute assigned as PRODUCT; this takes the product of the values of the OutputStates that it monitors (listed in its monitored_output_states attribute. However, this can be customized in a variety of ways:

  • by specifying a different function for the ObjectiveMechanism (see Objective Mechanism Examples for an example);
  • using a list to specify the OutputStates to be monitored (and the tuples format to specify weights and/or exponents for them) in either the monitor_for_control or objective_mechanism arguments of the LVOCControlMechanism’s constructor;
  • using the monitored_output_states argument for an ObjectiveMechanism specified in the objective_mechanism argument of the LVOCControlMechanism’s constructor;
  • specifying a different ObjectiveMechanism in the objective_mechanism argument of the LVOCControlMechanism’s constructor.

Note

If a constructor for an ObjectiveMechanism is used for the objective_mechanism argument of the LVOCControlMechanism’s constructor, then the default values of its attributes override any used by the LVOCControlMechanism for its objective_mechanism. In particular, whereas an LVOCControlMechanism uses the same default function as an ObjectiveMechanism (LinearCombination), it uses PRODUCT rather than SUM as the default value of the operation attribute of the function. As a consequence, if the constructor for an ObjectiveMechanism is used to specify the LVOCControlMechanism’s objective_mechanism argument, and the operation argument is not specified, SUM rather than PRODUCT will be used for the ObjectiveMechanism’s function. To ensure that PRODUCT is used, it must be specified explicitly in the operation argument of the constructor for the ObjectiveMechanism (see 1st example under Specifying Control for a System).

The LVOCControlMechanism’s function learns to predict the value of the OUTCOME OutputState of the LVOCControlMechanism’s objective_mechanism, as described below.

Function¶

The function of an LVOCControlMechanism learns how to weight its predictors, the values of its control_signals, the interactions between these, and the costs of the control_signals, to best predict the outcome of its objective_mechanism. Using those weights, and the current set of predictors, it then searches for and returns the allocation_policy that maximizes the EVC. By default, function is BayesGLM. However, any function can be used that accepts a 2d array, the first item of which is an array of scalar values (the prediction terms) and the second that is a scalar value (the outcome to be predicted), and returns an array with the same shape as the LVOCControlMechanism’s allocation_policy.

Note

The LVOCControlMechanism’s function is provided the values of the predictors and outcome of its objective_mechanism from the previous trial to update the prediction_weights `prediction_weights. Those are then used to determine (and implement) the allocation_policy that is predicted to generate the greatest EVC based on the predictor_values for the current trial.

ControlSignals¶

The OutputStates of an LVOCControlMechanism (like any Control Mechanism) are a set of ControlSignals, that are listed in its control_signals attribute (as well as its output_states attribute). Each ControlSignal is assigned a ControlProjection that projects to the ParameterState for a parameter controlled by the LVOCControlMechanism. Each ControlSignal is assigned an item of the LVOCControlMechanism’s allocation_policy, that determines its allocation for a given TRIAL of execution. The allocation is used by a ControlSignal to determine its intensity, which is then assigned as the value of the ControlSignal’s ControlProjection. The value of the ControlProjection is used by the ParameterState to which it projects to modify the value of the parameter (see Modulation for description of how a ControlSignal modulates the value of a parameter it controls). A ControlSignal also calculates a cost, based on its intensity and/or its time course. The cost may be included in the evaluation carried out by the LVOCControlMechanism’s function for a given allocation_policy, and that it uses to adapt the ControlSignal’s allocation in the future.

Execution¶

When an LVOCControlMechanism is executed, it uses the values of its predictors, listed in its predictor_values attribute, to determines and implement the allocation_policy for the current trial of execution of its composition. Specifically it executes the following steps:

  • Updates prediction_vector with the current predictors_values, control_signals, and their costs.
  • Calls its function with the prediction_vector and the outcome received from the LVOCControlMechanism’s objective_mechanism, discounted by the costs associated with each of its control_signals, to update its prediction_weights.
  • Calls its gradient_ascent function with prediction_vector and prediction_weights to determine the allocation_policy that yields the greatest EVC, and returns that allocation_policy.

The values specified by the allocation_policy returned by the LVOCControlMechanism’s function are assigned as the values of its control_signals.

predictor_values¶

1d ndarray – the current values of the InputStates used by function to determine allocation_policy (see Predictors for details about predictors).

objective_mechanism¶

ObjectiveMechanism – the ‘ObjectiveMechanism’ used by the LVOCControlMechanism to evaluate the performance of its system. If a list of OutputStates is specified in the objective_mechanism argument of the LVOCControlMechanism’s constructor, they are assigned as the monitored_output_states attribute for the objective_mechanism (see LVOCControlMechanism_ObjectiveMechanism for additional details).

monitored_output_states¶

List[OutputState] – list of the OutputStates monitored by objective_mechanism (and listed in its monitored_output_states attribute), and used to evaluate the performance of the LVOCControlMechanism’s system.

monitored_output_states_weights_and_exponents¶

List[Tuple[scalar, scalar]] – a list of tuples, each of which contains the weight and exponent (in that order) for an OutputState in monitored_outputStates, listed in the same order as the outputStates are listed in monitored_outputStates.

prediction_vector¶

1d ndarray – current values, respectively, of predictors, interaction terms for predictors x control_signals, control_signals, and costs of control_signals.

prediction_weights¶

1d ndarray – weights assigned to each term of prediction_vector last returned by function.

function¶

LearningFunction or callable – takes prediction_vector and outcome and returns an updated set of prediction_weights (see Function for additional details).

update_rate¶

int or float – determines the amount by which the value of each ControlSignal in the allocation_policy is modified in each iteration of the gradient_ascent method.

convergence_criterion¶

int or float – determines the change in estimate of the EVC below which the gradient_ascent method should terminate and return an allocation_policy.

max_iterations¶

int – determines the maximum number of iterations gradient_ascent method is allowed to execute; if exceeded, a warning is issued, and the method returns the last allocation_policy evaluated.

allocation_policy¶

2d np.array : defaultControlAllocation – determines the value assigned as the variable for each ControlSignal and its associated ControlProjection. Each item of the array must be a 1d array (usually containing a scalar) that specifies an allocation for the corresponding ControlSignal, and the number of items must equal the number of ControlSignals in the LVOCControlMechanism’s control_signals attribute.

control_signals¶

ContentAddressableList[ControlSignal] – list of the LVOCControlMechanism’s ControlSignals, including any that it inherited from its system (same as the LVOCControlMechanism’s output_states attribute); each sends a ControlProjection to the ParameterState for the parameter it controls

name¶

str – the name of the LVOCControlMechanism; if it is not specified in the name argument of the constructor, a default is assigned by MechanismRegistry (see Naming for conventions used for default and duplicate names).

add_predictors(predictors)¶

Add InputStates and Projections to LVOCControlMechanism for predictors used to predict outcome

predictors argument can use any of the forms of specification allowed for InputState(s),
as well as a dictionary containing an entry with SHADOW_EXTERNAL_INPUTS as its key and a list of ORIGIN Mechanisms and/or their InputStates as its value.
gradient_ascent(control_signals, prediction_vector, prediction_weights)¶

Determine the allocation_policy that maximizes the EVC.

Iterate over prediction_vector; for each iteration:

  • compute gradients based on current control_signal values and their costs (in prediction_vector);
  • compute new control_signal values based on gradients;
  • update prediction_vector with new control_signal values and the interaction terms and costs based on those;
  • use prediction_weights and updated prediction_vector to compute new EVC.

Continue to iterate until difference between new and old EVC is less than convergence_criterion or number of iterations exceeds max_iterations.

Return control_signals field of prediction_vector (used by LVOCControlMechanism as its allocation_vector).
